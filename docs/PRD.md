## 1. 제품 개요

### 1-1. 제품 정의

TicketTodo는 **단일 사용자용 칸반 보드 티켓 관리 앱**이다.  
로그인 없이 즉시 사용 가능하며, Backlog → TODO → In Progress → Done 4단계 흐름으로 업무를 시각적으로 관리한다.  
드래그 앤 드롭, 마감일 임박 경고, 기한 필터를 핵심 인터랙션으로 제공한다.

### 1-2. 제품 목표

| 목표             | 설명                                              |
| ---------------- | ------------------------------------------------- |
| 즉시 사용성      | 계정 생성·로그인 없이 앱 진입 즉시 티켓 생성 가능 |
| 시각적 상태 관리 | 4개 칼럼 칸반으로 업무 흐름 한눈에 파악           |
| 기한 관리        | D-3 이내·초과 티켓 색상 경고로 놓치는 업무 방지   |
| 직관적 조작      | DnD로 상태 전환, 클릭으로 상세 수정               |

### 1-3. 범위 경계

| 포함 (MVP)           | 제외 (Post-MVP)    |
| -------------------- | ------------------ |
| 티켓 CRUD            | 검색 기능          |
| 4단계 칸반 보드      | 우선순위 필터      |
| 드래그 앤 드롭       | 다중 사용자 / 협업 |
| 기한 임박 시각 경고  | 알림 / 푸시        |
| 이번주·기한초과 필터 | 통계 / 대시보드    |
| 모바일 반응형        | 오프라인 지원      |

## 2. 사용자 페르소나

### 페르소나: 준형 (단일 사용자)

> TicketTodo는 단일 사용자 앱이므로 페르소나는 1개다.

| 항목            | 내용                                                               |
| --------------- | ------------------------------------------------------------------ |
| 이름            | 준형                                                               |
| 역할            | 개인 업무·사이드 프로젝트를 혼자 관리하는 실무자                   |
| 환경            | 데스크탑(메인), 모바일(이동 중 확인)                               |
| 주요 Pain Point | 할 일이 흩어져 있고 마감 임박 업무를 자주 놓침                     |
| 기대 가치       | 가볍고 빠른 칸반. 설정 없이 켜자마자 쓸 수 있어야 함               |
| 사용 패턴       | 아침에 TODO 확인·재배치, 업무 중 DnD로 상태 변경, 저녁에 Done 확인 |

## 3. 핵심 사용자 시나리오

### 시나리오 A: 새 티켓 생성 후 칸반 배치

```
[목표] 새 업무를 등록하고 진행 상태를 설정한다
```

| 단계 | 사용자 행동                                                     | 시스템 반응                                              | 연관 FR         |
| ---- | --------------------------------------------------------------- | -------------------------------------------------------- | --------------- |
| 1    | 우측 상단 "새 업무" 버튼 클릭                                   | 티켓 생성 폼 모달 오픈                                   | FR-001          |
| 2    | 제목 입력 (필수), 설명·우선순위·시작예정일·종료예정일 선택 입력 | 실시간 유효성 검증 (제목 비어있으면 저장 비활성화)       | FR-001, NFR-016 |
| 3    | 저장 클릭                                                       | POST /api/tickets 호출 → Backlog 칼럼 최하단에 카드 추가 | FR-001          |
| 4    | 카드가 Backlog에 표시됨                                         | 제목·설명·우선순위 뱃지·dueDate 뱃지 렌더링              | FR-015          |
| 5    | 카드를 TODO 칼럼으로 DnD                                        | status=TODO, order 재계산 후 PATCH /api/tickets/:id      | FR-009, FR-011  |

**예외 흐름**
- 제목 미입력 저장 시도 → UI 단 에러 메시지 표시, API 호출 없음 (NFR-016)
- API 500 에러 → 토스트 메시지 노출, 카드 상태 롤백 (NFR-025, FR-011)

## 4. 화면별 동작 정의

### 4-1. 전체 레이아웃 구조

```
┌─────────────────────────────────────────────────────────┐
│  Header: [Search (Post-MVP)]          [새 업무 버튼]     │
├──────────────┬──────────────────────────────────────────┤
│              │  [이번주 업무] [일정이 초과된 업무]        │
│   Backlog    ├────────────┬─────────────┬───────────────┤
│  (좌측 고정  │    TODO    │ In Progress │     Done      │
│    패널)     │            │             │               │
│              │            │             │               │
└──────────────┴────────────┴─────────────┴───────────────┘
```

> 와이어프레임 기준: Backlog는 좌측 고정 패널, 나머지 3칼럼은 메인 칸반 영역 (FR-006)
