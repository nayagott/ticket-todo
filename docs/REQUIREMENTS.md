# REQUIREMENTS.md — TicketTodo MVP

> 최종 확정일: 2026-06-11
> 버전: v1.1

## 1. 프로젝트 개요

| 항목        | 내용                                                                                                     |
| ----------- | -------------------------------------------------------------------------------------------------------- |
| 앱명        | TicketTodo                                                                                               |
| 유형        | 단일 사용자 칸반 보드 (인증 없음)                                                                        |
| 핵심 엔티티 | ticket                                                                                                   |
| 칼럼 구성   | 백로그(Backlog) \| 할 일(TODO) \| 진행 중(In Progress) \| 완료(Done)                                    |
| 기술 스택   | Next.js App Router + API Routes, Drizzle ORM, Vercel Postgres, Tailwind CSS 4, @dnd-kit/core, TypeScript |
| 배포 환경   | Vercel + Vercel Postgres                                                                                 |
| MVP 범위    | 단일 사용자, 협업/인증 없음                                                                              |

## 2. 핵심 데이터 모델

### Ticket 필드

| 필드명      | 타입                                          | 필수 | 설명                          |
| ----------- | --------------------------------------------- | ---- | ----------------------------- |
| id          | uuid                                          | ✅    | PK                            |
| title       | string (max 255)                              | ✅    | 티켓 제목                     |
| description | text                                          | —    | 티켓 설명                     |
| priority    | enum (Low \| Medium \| High)                  | —    | 우선순위                      |
| status      | enum (Backlog \| TODO \| In Progress \| Done) | ✅    | 현재 칼럼 위치 (내부 DB 값은 영문, UI 표시명은 한국어) |
| order       | integer                                       | ✅    | 칼럼 내 순서 (1000 단위 간격) |
| startedAt   | timestamp                                     | —    | 시작예정일                    |
| dueDate     | timestamp                                     | —    | 종료예정일                    |
| createdAt   | timestamp                                     | ✅    | 생성 시각 (자동)              |
| updatedAt   | timestamp                                     | ✅    | 수정 시각 (자동)              |

## 3. 기능 요구사항 (FR)

### 3-1. 티켓 CRUD

| ID     | 기능명         | 설명                                                                                     | API 엔드포인트            |
| ------ | -------------- | ---------------------------------------------------------------------------------------- | ------------------------- |
| FR-001 | 티켓 생성      | 제목(필수), 설명, 우선순위, 시작예정일, 종료예정일 입력해 티켓 생성. 기본 칼럼은 Backlog | `POST /api/tickets`       |
| FR-002 | 티켓 목록 조회 | 전체 티켓을 칼럼별로 조회                                                                | `GET /api/tickets`        |
| FR-003 | 티켓 단건 조회 | 특정 티켓 상세 정보 조회                                                                 | `GET /api/tickets/:id`    |
| FR-004 | 티켓 수정      | 제목, 설명, 우선순위, 시작예정일, 종료예정일 수정                                        | `PATCH /api/tickets/:id`  |
| FR-005 | 티켓 삭제      | 티켓 단건 삭제. 삭제 전 확인 다이얼로그 필수                                             | `DELETE /api/tickets/:id` |

### 3-2. 칸반 보드

| ID     | 기능명             | 설명                                                                                                                 | API 엔드포인트           |
| ------ | ------------------ | -------------------------------------------------------------------------------------------------------------------- | ------------------------ |
| FR-006 | 보드 렌더링        | 백로그 / 할 일 / 진행 중 / 완료 4개 칼럼 수평 배치 렌더링 (내부 status 값: Backlog / TODO / In Progress / Done). 백로그는 좌측 고정 패널, 나머지 3개는 메인 칸반 영역 | — (클라이언트)           |
| FR-007 | 칼럼별 티켓 그룹핑 | `status` 필드 기준으로 티켓을 해당 칼럼에 배치                                                                       | — (클라이언트)           |
| FR-008 | 칼럼 내 순서 관리  | 같은 칼럼 내 티켓 간 순서(`order`) 유지 및 변경                                                                      | `PATCH /api/tickets/:id` |

### 3-3. 드래그 앤 드롭

| ID     | 기능명               | 설명                                                         | API 엔드포인트           |
| ------ | -------------------- | ------------------------------------------------------------ | ------------------------ |
| FR-009 | 칼럼 간 드래그앤드롭 | 티켓을 다른 칼럼으로 드롭 시 `status` 변경                   | `PATCH /api/tickets/:id` |
| FR-010 | 칼럼 내 순서 변경    | 같은 칼럼 내에서 드래그앤드롭으로 순서 재정렬                | `PATCH /api/tickets/:id` |
| FR-011 | 낙관적 업데이트      | 드롭 완료 즉시 UI 반영, API 실패 시 드롭 전 상태로 자동 롤백 | — (클라이언트)           |

### 3-4. 기간 임박 시각 표시

| ID     | 기능명          | 설명                                                                 | API 엔드포인트 |
| ------ | --------------- | -------------------------------------------------------------------- | -------------- |
| FR-012 | D-day 임박 경고 | `dueDate` 기준 D-3 이내(Done 제외) 티켓 카드에 주황색 테두리 표시    | — (클라이언트) |
| FR-013 | 기한 초과 경고  | `dueDate`가 오늘 이전이고 Done이 아닌 티켓 카드에 빨간색 테두리 표시 | — (클라이언트) |
| FR-014 | 정상 상태 표시  | D-4 이후 티켓은 기본 테두리(무채색) 표시                             | — (클라이언트) |

### 3-5. 티켓 카드 UI

| FR-016 | 티켓 상세 모달   | 카드 클릭 시 모달 열림 — 전체 필드 표시 및 인라인 수정 가능 | — (클라이언트) |

## 4. 비기능 요구사항 (NFR)

### 4-1. 성능

| ID      | 항목              | 핵심 기준                                               |
| ------- | ----------------- | ------------------------------------------------------- |
| NFR-001 | API 응답 시간     | 모든 REST API 응답 300ms 이하 (p95 기준)                |
| NFR-002 | 초기 화면 로드    | FCP(First Contentful Paint) 2초 이내 (Vercel Edge 기준) |
| NFR-003 | DnD 인터랙션 응답 | 드래그 시작 → UI 반응 16ms 이내 (60fps 유지)            |
| NFR-004 | 티켓 렌더링 한계  | 칼럼당 최대 100개 티켓까지 레이아웃 깨짐 없이 렌더링    |

### 4-2. 반응형

| ID      | 항목                 | 핵심 기준                                                                      |
| ------- | -------------------- | ------------------------------------------------------------------------------ |
| NFR-005 | 지원 해상도 범위     | 360px(모바일 최소) ~ 1920px(데스크탑 최대)                                     |
| NFR-006 | 모바일 칸반 레이아웃 | 768px 미만: 칼럼 가로 스크롤로 4칼럼 모두 접근 가능                            |
| NFR-007 | 터치 DnD             | 모바일/태블릿 터치 이벤트로 드래그앤드롭 동작 (@dnd-kit/core 터치 센서 활성화) |

### 4-3. 접근성

| ID      | 항목              | 핵심 기준                                                                   |
| ------- | ----------------- | --------------------------------------------------------------------------- |
| NFR-008 | 키보드 네비게이션 | Tab/Shift+Tab으로 모든 카드·버튼 포커스 이동 가능                           |
| NFR-009 | 키보드 DnD        | Space(집기) → 방향키(이동) → Enter(놓기)로 칼럼 간 이동 가능                |
| NFR-010 | 스크린리더 지원   | 카드 상태 변경 시 `aria-live`로 결과 음성 출력 (예: "할 일(으)로 이동됐습니다.")        |
| NFR-011 | 시맨틱 마크업     | 칼럼은 `role="list"`, 카드는 `role="listitem"`, 버튼은 `<button>` 태그 사용 |
| NFR-012 | 색상 대비         | 임박/초과 테두리 색상 WCAG AA 기준 대비율 4.5:1 이상                        |

### 4-4. 데이터 정합성

| ID      | 항목             | 핵심 기준                                                                              |
| ------- | ---------------- | -------------------------------------------------------------------------------------- |
| NFR-013 | 낙관적 업데이트  | DnD 드롭 즉시 UI 반영, API 실패 시 드롭 전 상태로 자동 롤백                            |
| NFR-014 | UI-DB 이중 검증  | API 응답 후 서버 반환값으로 클라이언트 상태 재동기화. 응답값 ≠ 로컬 상태면 서버값 우선 |
| NFR-015 | order 충돌 방지  | order 필드 정수 1000 단위 간격 부여, 삽입 시 중간값 계산으로 재정렬 최소화             |
| NFR-016 | 입력값 이중 검증 | 제목 필수·최대 255자, dueDate 형식 — UI 단 + API 단 각각 독립 검증                     |
| NFR-017 | 삭제 방식        | 하드 삭제. 삭제 전 확인 다이얼로그 필수                                                |
